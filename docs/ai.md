# AI設計

## 1. 目的と基本方針

本AIは、置き菓子事業者が各設置場所に対して次回補充・出荷する商品構成を決めるための支援機能である。

設計方針として、単純なルールベースではなく **Predict-then-Optimize** のフレームワークを採用する。

- Predict: 設置場所ごと・商品ごとの次回補充期間における需要を予測する
- Optimize: 予測需要、在庫、設置容量、商品ステータスなどの制約を元に、推奨商品と推奨数量を最適化する

MVPでは、AIが商品構成を完全自動で確定するのではなく、担当者が推奨結果を確認し、必要に応じて調整・確定する運用を前提とする。

## 2. 入力

AIの入力は、需要予測に使うデータと、最適化に使う制約データに分ける。

### 需要予測に使う入力

- 購入データ
  - 購入日時
  - 企業ID
  - 設置場所ID
  - 商品ID
  - 数量
  - 金額
  - 決済ステータス
- 商品マスタ
  - 商品ID
  - 商品名
  - 販売価格
  - 原価
  - カテゴリ
  - 販売ステータス
- 導入企業・設置場所情報
  - 企業ID
  - 所在地域
  - 設置場所ID
  - 補充頻度
  - 最大設置可能数量

需要予測では、成功した購入データのみを販売実績として扱う。失敗、キャンセル、返金は原則として販売実績から除外する。

### 最適化に使う入力

- 予測需要
  - 設置場所ID
  - 商品ID
  - 予測販売数量
  - 予測期間
- 在庫データ
  - 商品ID
  - 保管在庫数
  - 設置場所ごとの現在設置数
  - 最終補充日時
  - 最終棚卸日時
- 制約・運用設定
  - 設置場所の最大設置可能数量
  - 商品ごとの最低数量
  - 商品ごとの最大数量
  - 販売停止商品の除外
  - 新商品・戦略商品の優先度
  - 除外商品

## 3. 出力

AIは、設置場所ごとに次回補充・出荷の推奨結果を出力する。

### 推奨結果

- 設置場所ID
- 商品ID
- 商品名
- 推奨数量
- 予測販売数量
- 推奨対象に含めた理由
- 制約適用結果
- 注意フラグ

### 注意フラグの例

- 在庫不足
- 販売停止
- 予測需要が低い
- 予測需要が高い
- 新商品
- 戦略商品
- 設置容量による数量制限

### 担当者操作後に保存する値

- AI推奨数量
- 担当者による調整後数量
- 確定数量
- 調整理由
- 確定ステータス

AIの出力は、担当者がそのまま採用するだけでなく、画面上で数量変更、商品追加、商品除外を行える前提とする。

## 4. 使用するモデル

本AIは Predict（需要予測）と Optimize（商品構成最適化）の2モデルを組み合わせる。MVPでは予測の点推定を最適化へ渡す逐次型 (two-stage) を採用し、改善フェーズで意思決定指向 (decision-focused) への発展を視野に入れる。各フェーズの全体像は本章末の導入ステップ表に集約する。

- 逐次型 (two-stage): 予測の点推定を最適化の入力に使う。実装が単純で運用しやすいMVPの構成
- 意思決定指向 (decision-focused): 予測誤差ではなく最終的な意思決定の損失を最小化するよう予測モデルを学習する。SPO+ や微分可能最適化レイヤーが代表手法で、「5. 実行履歴と評価」で蓄積する推奨・確定・実績の差分が教師信号になる

### Predict: 需要予測モデル

設置場所ごと・商品ごとの次回補充期間における販売数量を予測する。テーブルデータに強い勾配ブースティング木を基本とする。

- 候補モデル: LightGBM（MVPの第一候補）、CatBoost、XGBoost
- 予測対象: 次回補充期間における商品別販売数量
- 学習単位: 設置場所ID × 商品ID × 集計期間
- 主な特徴量: 直近販売数量、過去一定期間の平均販売数量、販売価格、カテゴリ、所在地域、補充頻度、曜日・月・季節、新商品フラグ、戦略商品フラグ

設計上の主要論点と対応方針は以下のとおり。

- 間欠需要: 設置場所×商品の粒度では販売ゼロの期間が多く、二乗誤差回帰は過小予測に偏る。LightGBM を Poisson / Tweedie 損失で学習し、Croston 法 / SBA をベースライン比較に用いる
- 不確実性: 点推定に加え予測区間を扱い、後述の欠品/売れ残りコスト最適化に接続する。分位点回帰で予測区間を得て、将来は NGBoost 等の分布回帰へ
- コールドスタート: 実績の乏しい新商品・新設置場所は、商品特徴量や設置場所属性から類似対象の実績を借用し、クラスタ平均をフォールバックとする
- カニバリ・代替: 独立予測では「商品Aを置くと商品Bの売上が食われる」効果を捉えられない。改善フェーズで離散選択モデル（MNL）を導入し、品揃え集合に対する各商品の選択確率をモデル化する

### Optimize: 商品構成最適化モデル

予測モデルの出力を元に、設置場所ごとの推奨商品と推奨数量を決定する。OR におけるアソートメント最適化に相当し、MVPでは混合整数線形計画法 (MILP) を用いる。

- 手法: Integer Programming / Mixed Integer Linear Programming
- ソルバー: OR-Tools (CP-SAT) を第一候補とし、規模・性能要件次第で Gurobi 等の商用ソルバーを検討
- 目的関数の候補: 予測販売数量／売上／粗利の最大化、欠品リスク・売れ残りリスクの低減
- 制約条件: 保管在庫数・設置最大数量を超えない、販売停止商品を含めない、商品ごとの最低/最大数量、新商品・戦略商品の優先度

最適化の定式化は段階的に拡張する。

- 決定論的 MILP（MVP）: 点推定の予測需要を所与とし、容量・在庫制約下で「予測販売数量をできるだけ満たす商品構成」を求める
- Newsvendor 型: 予測分布を用いて欠品の機会損失と売れ残りコストを天秤にかける。賞味期限・回収コストのある置き菓子と相性が良い
- 選択モデル下のアソートメント最適化: MNL 前提で「どの商品を置くか」という集合選択自体を最適化し、カニバリを考慮する
- ロバスト / 確率計画: 需要の不確実性を制約・目的に明示的に織り込む（将来オプション）

### 段階的な導入ステップ

| フェーズ | Predict | Optimize |
| --- | --- | --- |
| MVP | LightGBM（Poisson/Tweedie 損失）で点推定 | 決定論的 MILP（OR-Tools CP-SAT）で容量・在庫制約下の数量最適化 |
| 改善1 | 分位点回帰で予測区間を追加 | Newsvendor 型（欠品/売れ残りコスト）に目的関数を拡張 |
| 改善2 | 離散選択モデル（MNL）でカニバリ考慮 | 選択モデル下のアソートメント最適化 |
| 改善3 | 意思決定指向（SPO+ など）への移行 | 予測と最適化を統合した end-to-end 学習 |

## 5. 実行履歴と評価

AI実行結果は履歴として保存し、後から販売実績と比較できるようにする。

### 保存する履歴

- 実行ID
- 実行日時
- 対象企業ID
- 対象設置場所ID
- 入力データの対象期間
- 予測販売数量
- AI推奨商品
- AI推奨数量
- 担当者による調整後数量
- 確定数量
- 確定ステータス

### 評価する内容

- 予測販売数量と実績販売数量の差
- AI推奨数量と確定数量の差
- 欠品が発生したか
- 売れ残りが発生したか
- 担当者による修正が多い商品は何か

これらの履歴を蓄積し、需要予測モデルの再学習や、最適化モデルの目的関数・制約条件の改善に利用する。

## 6. 用語集

本ドキュメントで使う、説明が必要な用語をまとめる。

### 点推定

予測値を「1つの数値」として出すこと。たとえば「次回の販売数量は8個」と言い切る形。手軽だが、「8個前後にどれくらいばらつくか（5〜12個の可能性があるのか、7〜9個に収まりそうなのか）」という不確実性の情報は持たない。これに対し、予測区間や予測分布は「どれくらいの確率でどの範囲に入るか」まで表現する。欠品と売れ残りのコストを天秤にかける最適化（Newsvendor 型）には、点推定だけでは不十分で、ばらつきの情報が必要になる。

### SPO+ (Smart Predict-then-Optimize)

逐次型 Predict-then-Optimize の弱点を補う学習手法。通常の予測モデルは「予測値が実績にどれだけ近いか（予測誤差）」を小さくするよう学習するが、予測誤差が小さくても最終的な商品構成の意思決定が良いとは限らない。SPO+ は「その予測を使って最適化した結果、どれだけ良い意思決定になったか」を直接の評価基準にして予測モデルを学習する。Elmachtoub と Grigas が提案した、この分野の代表的手法。

### 微分可能最適化レイヤー

最適化問題（MILP など）を、ニューラルネットワークの1つの層（レイヤー）として組み込めるようにした仕組み。通常、最適化の計算過程は「予測値を少し変えると最適解がどう変わるか」を勾配として取り出せず、予測モデルと最適化を一体で学習できない。微分可能最適化レイヤーはこの勾配を計算可能にし、予測から最適化までを end-to-end（端から端まで一気通貫）で学習することを可能にする。OptNet などが代表例。
